## Context & motivation

CUBE exec and senior members needed a structured way to file, track, and manage strikes
against members. Previously there was no in-portal tooling â€” strikes were tracked manually.
This adds a full strike workflow with role-based access, exec approval, and automated email
notifications through Resend.

## What changed

- Added `/portal/strikes` dashboard with three role-specific views: exec sees all strikes
  (pending queue + history), filers see what they've submitted, members see strikes against them
- Added `/portal/strikes/new` submission page â€” execs file and auto-approve in one step;
  project managers and senior consultants submit requests that go to the pending queue
- Added `/portal/strikes/[id]` detail/approval page where execs can approve, deny, void,
  downgrade (full â†’ half), or upgrade (half â†’ full) a strike
- Added `EmailEditorModal` so execs can review and edit outgoing notification emails before
  sending â€” templates are pre-filled but fully editable
- Added `MemberSearch` autocomplete component backed by a new `GET /api/members/search` route
- Added API routes: `GET/POST /api/strikes`, `GET /api/strikes/[id]`,
  `POST /api/strikes/[id]/resolve` (exec-only)
- Added `lib/email/strikes.ts` with email templates for all outcomes (approval, denial, void,
  downgrade, upgrade) sent to both the struck member and the original requester
- Auto-alerts the full exec team via email when a member reaches 3 or more effective strikes
- Added `resend` package for transactional email delivery

2. `npm run dev`
3. Sign in as an exec-role member and go to `/portal/strikes`
   - Verify the full strike list renders with pending queue at the top
   - Click "File a strike", search for a member, submit â€” confirm it appears as approved immediately
   - Click into a strike and verify the approve/deny/void/downgrade/upgrade actions appear
   - Confirm `EmailEditorModal` opens with pre-filled content before the action fires
4. Sign in as a project manager or senior consultant
   - Go to `/portal/strikes/new` and submit a request â€” confirm it lands as `pending`
   - Verify non-exec users cannot access `/portal/strikes/[id]` resolve actions
5. Sign in as a regular consultant and verify `/portal/strikes` shows only strikes against that member
6. Confirm `RESEND_API_KEY` is set in `.env.local`; without it, email sends will silently fail

Tested on: Chrome, macOS, desktop viewport

## Breaking changes & rollback

Requires a `strikes` table in Supabase with the columns referenced in the API routes
(`member_id`, `filed_by`, `strike_type`, `effective_type`, `status`, `reason`,
`resolved_at`, `resolved_by`). If the table does not exist, all `/api/strikes` routes will
500. Rollback: `git revert ca0b56c f9ba80f` and remove the `resend` dependency.

`RESEND_API_KEY` must be added to environment â€” no other env changes required.
